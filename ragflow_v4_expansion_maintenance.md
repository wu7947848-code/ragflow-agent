# RAGFlow v4 知识库扩展维护规范

本文档用于记录南京邮电大学政策类智能检索助手后续扩展新数据时的维护原则。当前保研/推免助手的架构可以沿用到其他学校官网政策领域，但新增资料不能直接裸加入知识库，必须经过准入、结构化、测试和回归。

## 1. 可复用的总体架构

推荐继续沿用当前 v4 架构：

```text
领域识别
→ 安全/越界拦截
→ 是否需要澄清
→ 会话状态抽取
→ 领域专属高风险路由
→ 领域事实库检索
→ 官方原文库检索
→ 受控回答
```

核心思想：

- 官方原文库负责提供原始依据。
- 结构化事实卡负责提供边界、索引、高风险精确信息。
- 确定性路由负责处理容易出错、不能交给普通 RAG 自由发挥的问题。
- 测试集负责防止新增资料破坏既有能力。

## 2. 新增资料准入清单

每次新增资料前，必须先记录以下信息：

```text
资料名称：
来源网址：
发布日期：
适用年份：
适用对象：
政策端/业务端：
是否官方原文：
是否历史材料：
是否可用于当前回答：
与现有资料是否可能冲突：
备注：
```

准入原则：

- 未知来源资料不得直接进入官方原文库。
- 历史材料必须明确标注年份和历史属性。
- 学院级材料不得扩展成全校规则。
- 校级材料不得替代学院具体安排。
- 没有明确依据的信息不得通过模型估算、补全或按往年推断。

## 3. 事实卡维护规则

新增一个领域时，至少应补充对应事实卡。事实卡不等于官方政策原文，只是检索增强和边界控制材料。

事实卡应覆盖：

- 高频真实问法。
- 精确联系方式、时间、地点、邮箱。
- 年份边界。
- 校级规则和学院规则的适用范围。
- 易混概念。
- 缺失信息和未明示信息。
- 历史材料不得外推的提醒。

事实卡建议命名：

```text
9x_facts_<domain>_<topic>.md
```

例如：

```text
98_facts_scholarship_scope.md
99_facts_transfer_major_requirements.md
```

事实卡中必须写明：

```text
文件性质：结构化事实表，不是官方政策原文。
来源文件：
适用年份：
适用范围：
不得推断项：
```

## 4. 冲突处理优先级

新增资料后，如果不同材料之间存在不一致，按以下顺序处理：

1. 同一事项优先使用最新官方正式通知。
2. 学院具体事项优先使用该学院当年细则。
3. 校级资格条件优先使用学校管理办法或校级正式通知。
4. 操作流程优先使用当年系统通知或操作指南。
5. 历史材料只能作为历史参考，不得作为当前政策结论。
6. 无明确官方依据时，必须回答“本知识库未收录明确规定”，不得自行推断。

典型冲突场景：

- 2025届材料和2026届材料冲突。
- 校级通知和学院细则表述不同。
- 官网新闻稿和正式通知不同。
- 同一学院存在新旧版本文件。
- 不同学院材料提交邮箱、复试时间、要求不同。

处理原则：

- 不合并不同学院规则。
- 不把接收端规则用于推荐端。
- 不把推荐端资格条件用于接收端复试录取。
- 不把历史年份材料套用到当前年份。

## 5. 路由规则新增标准

只有满足以下条件之一，才新增确定性路由：

- 普通 RAG 多次出现稳定错误。
- 问题涉及邮箱、电话、时间、年份、资格边界等高风险精确信息。
- 问题涉及“能否推断”“按往年”“帮我估计”等不应回答的请求。
- 问题涉及推荐端/接收端混用。
- 问题涉及无关、恶意、越权或诱导泄露内部配置。

不建议为每个普通问法都加路由。普通问题仍应优先由事实库和官方原文库检索解决。

路由新增原则：

- 规则越窄越好。
- 更具体的路由放在更泛化的路由前。
- 新增路由必须配套测试。
- 固定回答必须包含“答案、依据、适用范围、提醒”四个部分。

## 6. 测试集分层

以后新增资料时，不要只依赖人工试问。建议按以下层级维护测试：

```text
基础可答测试
边界/年份测试
连续追问测试
恶意/无关问题测试
高风险事实测试
新增资料专项测试
全量回归测试
```

每个新领域至少应新增：

- 5 到 10 个真实学生问法。
- 3 个年份/历史边界问题。
- 3 个推荐端/接收端或业务边界问题。
- 2 个连续追问问题。
- 2 个无依据/要求推断问题。

测试原则：

- 断言要检查关键事实和禁止项。
- 不要因为同义表达不同而误判失败。
- 对邮箱、电话、日期、比例、年份等精确信息必须严格断言。
- 每次新增事实卡或路由后，先跑专项测试，再跑核心回归。

## 7. 推荐扩展流程

新增资料时按以下流程执行：

```text
1. 收集资料
2. 填写资料准入清单
3. 判断是否官方原文、历史材料、当前有效材料
4. 加入官方原文库或历史参考区
5. 更新/新增结构化事实卡
6. 检查是否需要新增高风险路由
7. 增加专项测试
8. 跑专项测试
9. 跑核心回归
10. 同步到 RAGFlow
11. 做少量人工抽问
12. 记录本次变更
```

## 8. 不建议的做法

以下做法应避免：

- 把所有新资料直接丢进一个大知识库。
- 只靠 prompt 约束，不做事实卡和测试。
- 把历史材料和当前材料混在一起不标注。
- 为每个用户问法都堆固定规则。
- 不跑回归就同步到可用助手。
- 用一个学院的信息补另一个学院的缺失项。
- 用模型“猜一个合理答案”。

## 9. 当前 v4 的维护基线

当前保研/推免 v4 已形成以下基线：

- 主知识库：官方原文层。
- 事实知识库：结构化事实卡层。
- 工作流：安全拦截、不能推断、澄清、状态抽取、检索改写、高风险路由、双知识库检索、受控回答。
- 测试：已有全量 v4 回归脚本和压力测试脚本。

后续扩展新领域时，应尽量复用这套骨架，而不是另起一套完全不同的结构。
